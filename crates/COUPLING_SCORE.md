### Две «ручки» для настройки влияния веса зависимостей

(работают в выражении штрафа)

$$
\color{orange}{K_{\text{slope}}}\;\cdot\;\ln\!\Bigl(\dfrac{\bar S}{\color{cyan}{\alpha_{\!Ds}}}+1\Bigr),
\qquad
\bar S=\dfrac{Ds}{\max(1,\;Ce)}
$$

| Константа                          | Что делает                                                                                                                                                  | Как ощущается на графиках                                                                                                                                        |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`K_SLOPE`**<br>*крутизна*        | Множитель перед логарифмом. Чем больше, тем **резче** CAS падает, как только растёт средняя сила зависимостей `S̄`.                                         | • Контуры сгущаются, градиент «синий → жёлтый» становится круче.<br>• При одинаковом `Ds`-значении граница «опасной» области подтягивается к точке (Ca=0, Ce=0). |
| **`ALPHA_DS`**<br>*сдвиг (бюджет)* | Делит `S̄` внутри ln(). Это бесплатный «запас» веса зависимостей. Увеличив α, вы **сдвигаете** весь эффект вправо: прежний штраф начнётся при больших `Ds`. | • Вся «синяя» зона смещается вправо / вверх по Ce.<br>• Крутизна НЕ меняется, меняется только то, в какой точке она «включается».                                |

---

## Как подбирать

### 1. Настройка `ALPHA_DS` — «где начинается боль»

| Хотите…                                                          | …тогда                        | Пример                                                                                    |
| ---------------------------------------------------------------- | ----------------------------- | ----------------------------------------------------------------------------------------- |
| Считать «лёгкими» и почти не штрафовать зависимости до суммы ≈ 6 | `ALPHA_DS = 6`                | Если `Ce = 3`, то `S̄ = Ds/Ce = 2`; делим на 6 → 0.33 → ln(1.33) почти 0, штраф мизерный. |
| Строже относиться к любым зависимостям                           | `ALPHA_DS < 1` (например 0.5) | Даже `Ds = 1` даст ощутимый вклад, потому что `ln(S̄/0.5 + 1)` ≈ ln(3)                    |

### 2. Настройка `K_SLOPE` — «насколько больно»

| Хотите…                                            | …тогда              | Эффект                                                               |
| -------------------------------------------------- | ------------------- | -------------------------------------------------------------------- |
| Модель, где тяжесть зависимостей критична          | `K_SLOPE ↑` (18…24) | CAS резко идёт вниз уже при умеренном `Ds`.                          |
| Модель, в которой количество связей важнее их веса | `K_SLOPE ↓` (6…8)   | Лог-штраф сглаживается, главная роль переходит к линейному `6·Ctot`. |

## Мини-таблица ощущений

| `K_SLOPE` | `ALPHA_DS` | `Ds` при Ce=3, где CAS ≈ 60 % (Ca=5, без циклов) |
| --------- | ---------- | ------------------------------------------------ |
| 12        | 1          | **≈ 10**                                         |
| 12        | 2          | **≈ 20**                                         |
| 8         | 1          | **≈ 15**                                         |
| 8         | 2          | **≈ 30**                                         |

> **Запомнить просто:**
> • **`K_SLOPE`** — наклон (чем круче, тем больнее любая тяжёлая зависимость).
> • **`ALPHA_DS`** — горизонтальный сдвиг (чем больше, тем позже начинается боль).

### Что такое `ABS_MAX`

`ABS_MAX` — это **масштабирующая константа** для финального шага нормализации:

```rust
let cas_raw = 100.0 - (abs_score * 100.0) / ABS_MAX;
```

* `abs_score` растёт при «плохих» факторах.
* Когда `abs_score == ABS_MAX`, дробь = 100 ⇒ `cas_raw` падает ровно до 0 %.
* Если `abs_score` ещё выше, результат уходит в минус и сразу обрезается `clamp(0, 100)`.

Таким образом `ABS_MAX` задаёт **границу, где начинается абсолютно плохой (0 %) балл**.

## Откуда взялась 140?

*Мы взяли 140 как «удобное круглое» после быстрой калибровки маленького сета.*
Но в оригинальном FtaQl эта роль играет 171, а в вашем проекте нужное значение, вероятно, **другое**.

## Как подобрать `ABS_MAX`

| Метод                     | Алгоритм                                                                                                                         | Когда лучше                                                           |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| **Эмпирический «бросок»** | Прогоняете 30-50 реальных модулей, смотрите max(abs\_score) и округляете вверх (например, 195 → 200).                            | Небольшой проект, нет жёстких требований.                             |
| **Процентиль 95/99**      | Вычисляете abs\_score для всего репозитория, берёте 95-й (или 99-й) процентиль и округляете.                                     | Большие code-базы, хочется, чтобы «единичные ужасы» не сжимали шкалу. |
| **Авто-пересчёт**         | Храните `ABS_MAX` в конфиге и обновляете nightly: `ABS_MAX = p99(abs_scores)`.                                                   | CI-pipeline, динамический рост проекта.                               |
| **Фикс под SLA**          | Решаете: «CAS должен упасть до 0 %, когда Ce=50, Ca=0, Ds=200, 3 цикла». Считаете abs\_score в этой точке → ставите в `ABS_MAX`. | Требуется жёсткий инженерный допуск («хуже этого быть не должно»).    |

### Влияние на шкалу

| Сценарий                     | Что будет, если…                                                               |
| ---------------------------- | ------------------------------------------------------------------------------ |
| **`ABS_MAX` слишком низкий** | Показатель быстро «залипает» на 0; потеря различимости среди «плохих» модулей. |
| **`ABS_MAX` слишком высок**  | Почти все CAS окажутся 70–100 %, «красной зоны» может не остаться.             |

---

### Совет при изменении других коэффициентов

Любая правка `K_SLOPE`, `K_INSTAB`, `ALPHA_DS` меняет распределение `abs_score`.
После таких изменений **проверьте гистограмму** новых `abs_score`:

```text
abs_min … abs_95 … abs_max
```

и подровняйте `ABS_MAX`, чтобы:

* крайние 1-5 % действительно обрезались к 0 %;
* «нормальные» модули продолжали распределяться по шкале 40–90 %.

### Мини-рецепт

```rust
/// Перекалибровка ABS_MAX (один раз в CI):
let scores: Vec<f64> = all_modules
    .map(|m| compute_abs_score(m))  // та же формула, но без нормирования
    .collect();
let p99 = percentile(&scores, 99.0);  // возьмите готовую lib/stat
const ABS_MAX: f64 = (p99.ceil() / 10.0).ceil() * 10.0; // округление к десяткам
```

*Так вы автоматически держите шкалу в актуальном диапазоне при росте проекта.*

---

**Итого:**

* **`K_SLOPE`** и **`ALPHA_DS`** формируют сам рельеф штрафа.
* **`ABS_MAX`** лишь говорит: «Где обрезаем до нуля».
  Подберите её под свою статистику, чтобы CAS-шкала оставалась информативной.
